RDS — Relational Database Service

Managed relational database. AWS handles provisioning, patching, backups, monitoring, and hardware failure. Supports MySQL, PostgreSQL, MariaDB, Oracle, SQL Server, and Db2. You connect over standard DB protocols — but you cannot SSH into the underlying instance.

RDS vs DB on EC2 — why managed wins
What RDS handles for you

OS patching, DB engine patching, automated backups, point-in-time restore, Multi-AZ failover, read replica creation, storage auto-scaling, CloudWatch monitoring, parameter groups. You manage the schema and queries — AWS manages everything else.

What you give up vs EC2

No SSH access. Can't install custom OS-level software or agents. Can't use unsupported DB engines. RDS Custom gives some of this back for Oracle and SQL Server when you genuinely need OS access.

When EC2+DB makes sense

Unsupported DB engines, OS-level tuning requirements, very specific version pinning, or software that must co-locate with the DB. Rare — RDS wins operationally almost every time.

Q

A team debates EC2+PostgreSQL vs RDS PostgreSQL. What specifically does the team stop managing by choosing RDS?

A

With RDS: no more OS patching, DB engine upgrades, hardware provisioning, storage scaling, backup scheduling, Multi-AZ replication setup, read replica creation, or failover configuration. The team still manages: DB schema, users, query optimisation, parameter groups, security groups. The operational compound interest is enormous — no more 2am patching windows.

Exam framing: "Managed service", "less operational overhead", "undifferentiated heavy lifting" → RDS over EC2. Only choose EC2+DB when there's a specific requirement RDS cannot satisfy.
RDS Storage Auto Scaling
What it doesAutomatically increases EBS storage when free space runs low — no manual action, no downtime. You set a Maximum Storage Threshold as a cost ceiling.
Trigger conditionFree storage <10% of allocated AND this lasts >5 minutes AND 6+ hours since last modification.
Important caveatStorage only scales up — never down. A temporary data migration that balloons the DB leaves that storage allocated permanently. Plan the max threshold carefully.
Read Replicas vs Multi-AZ — the most tested distinction
Read Replicas — for performance (read scale-out)
App (writes)→ Primary (us-east-1a)async → Replica 1 (us-east-1b)
App (reads)→ Replica 2 (us-east-1c)
App (reads)→ Replica 3 (eu-west-1) — cross-region

Async replication → replicas may lag. Up to 5. Can be promoted to standalone. Not for failover — for read scaling. App must explicitly use replica endpoint.

Multi-AZ — for disaster recovery (auto failover)
App→ Primary (us-east-1a)sync → Standby (us-east-1b) — NOT readable
↓ primary fails → DNS flips automatically (~60s)
App (same endpoint)→ Standby promoted to primary

Sync replication — standby always in lockstep. One DNS endpoint AWS flips on failover. Standby is never readable — exists purely for DR.

Q

A production MySQL RDS serves a read-heavy analytics dashboard that's slowing down the main app. What do you add, and what app change is needed?

A

Create one or more Read Replicas. Point the dashboard's connection string to the replica endpoint. The main app continues writing to the primary. The app must explicitly use the replica endpoint — RDS does not auto-route reads. The replica absorbs dashboard queries; primary writes are unaffected.

The app change is mandatory. Adding a read replica alone changes nothing — your code must use the replica's endpoint for reads. A common exam distractor is implying that RDS automatically load-balances reads to replicas. It does not.
Q

A company wants to migrate RDS MySQL from us-east-1 to eu-west-1 with minimum downtime. How?

A

Create a cross-region Read Replica in eu-west-1. Once replication catches up, promote the replica to a standalone DB in eu-west-1. Update the app connection string. Schedule the promotion during low traffic to minimise the lag window. No data loss if done carefully.

Read replicas can be promoted to standalone primary DBs. This is the standard cross-region migration pattern. Once promoted, the replica is independent — no sync back to the original region.
Q

An RDS instance has both Multi-AZ and a Read Replica. The primary fails. What happens to each?

A

Multi-AZ: Standby is automatically promoted in ~60s. DNS endpoint stays the same — app reconnects automatically. Read Replica: Continues replicating, now from the promoted standby. No action needed on the replica.

They serve different purposes. Multi-AZ = automatic failover for resilience. Read Replica = read scaling. Use both in production: Multi-AZ for HA, read replicas for performance.
Replication cost — same-region vs cross-region Read Replica replication within the same region = free (even across AZs). Cross-region Read Replica replication = charged. Multi-AZ sync replication = always free. This appears in cost-optimisation questions.
RDS Custom
What it isRDS for Oracle and SQL Server with SSH/SSM access to the underlying EC2 instance. Install custom OS patches, monitoring agents, or licensing tools while keeping some RDS management.
Exam trigger"Need OS access on RDS", "custom OS patches", "install agents on RDS" → RDS Custom. Oracle and SQL Server only.

Aurora

AWS's proprietary cloud-native relational database. MySQL and PostgreSQL compatible. 5× MySQL throughput, 3× PostgreSQL throughput at similar cost. Storage auto-grows in 10 GB increments to 128 TB. Everything about Aurora is designed for HA and scale from the ground up.

Aurora High Availability — the storage layer is everything
Aurora storage — 6 copies across 3 AZs, always
Aurora Writer (Primary)→ writes to → Shared distributed storage (6 copies, 3 AZs)
Aurora Reader 1→ reads from → same shared storage (zero replication lag)
Aurora Reader 2→ reads from → same shared storage (zero replication lag)

6 copies across 3 AZs. Tolerates 2 copy losses without write impact, 3 copy losses without read impact. Self-healing — corrupted blocks replaced automatically. Write quorum = 4/6 copies.

Failover time<30 seconds — promotes a read replica to writer. Since all replicas share the same storage, there's no data to sync. Much faster than RDS Multi-AZ (~60s).
Read replicasUp to 15 (vs 5 for RDS). All read from the same shared storage — zero replication lag. Any can be promoted to writer.
Aurora GlobalOne primary region, up to 5 secondary regions. Cross-region replication <1 second lag. Secondary promotable to primary in <1 minute for global DR.
Aurora DB Cluster — Endpoints explained
One cluster, multiple connection endpoints
App (writes)→ Writer Endpoint — always points to current master
App (reads)→ Reader Endpoint — load balances across all replicas
Analytics team→ Custom Endpoint — specific subset (e.g. large r6g instances only)
Instance Endpoint — direct to one instance (avoid in prod)

Writer Endpoint auto-flips to new master on failover. Reader Endpoint does connection-level (not query-level) load balancing. Custom Endpoints enable workload isolation within the same cluster.

Q

Aurora has 3 small replicas for the app and 2 large replicas for analytics. Analytics queries are slow because they compete with app read traffic on the Reader Endpoint. Fix without a new cluster.

A

Create two Custom Endpoints: one pointing to the 2 large replicas (for analytics), one to the 3 smaller replicas (for the app). Each team connects to their custom endpoint. Workloads are isolated — analytics can run heavy queries without affecting app read latency, all within the same Aurora cluster.

Custom Endpoints = workload isolation without separate clusters. This saves the cost of a second Aurora cluster. A common exam scenario: "isolate analytics from app reads" → Custom Endpoint, not a new cluster.
Q

Aurora primary writer fails. What happens to the Writer Endpoint? Does the app need to change its connection string?

A

No change needed. Aurora promotes a read replica to writer in <30s and updates the Writer Endpoint DNS automatically. The app reconnects to the same URL — it's now talking to the new primary. This is the entire point of the Writer Endpoint: it abstracts away which physical instance is master.

Never hardcode Aurora instance endpoints in production. Always use Writer Endpoint for writes, Reader/Custom Endpoint for reads. Instance endpoints break on any topology change.
Aurora Serverless
What it isAurora that auto-scales compute up/down based on load — including scaling to zero when idle. Define min/max ACU (Aurora Capacity Unit) range. No instance to size or manage.
Connection proxyA managed proxy fleet sits in front. Your app connects to the proxy endpoint. Scaling happens behind it — connections are maintained through scale events.
Best forInfrequent or unpredictable workloads: dev/test, new apps with unknown traffic, multi-tenant SaaS with irregular usage, apps with long idle periods.
v1 vs v2v2 is current. Scales in fine 0.5 ACU increments, much faster scaling, supports Multi-AZ + read replicas + Global Database. Always use v2 for new workloads.
Q

A SaaS company has hundreds of small customers, each needing an isolated DB. Most are inactive 20+ hours/day. Provisioned Aurora per customer is too expensive. What's the solution?

A

Aurora Serverless v2 per customer. Scales to near-zero during inactivity — the idle DB costs almost nothing. When a customer logs in, Aurora scales up in seconds. You pay per ACU-second consumed, not for provisioned capacity sitting idle. For hundreds of sporadic DBs, the cost savings are dramatic vs provisioned.

Serverless = sporadic/unpredictable workloads. Provisioned = consistent/predictable. Provisioned charges 24/7 regardless of use. Serverless charges per-second of actual capacity. Brief scale-up latency is the trade-off.

Q

A developer accidentally ran DELETE FROM orders WHERE 1=1 on production Aurora MySQL. Restoring from backup would take hours. Faster option?

A

Use Aurora Backtrack. Rewind the cluster to a point just before the DELETE — takes minutes, not hours, and happens in-place (same cluster, no new instance). Configure a Backtrack window that covers enough time for your team to catch such mistakes (e.g. 24h). No backup restore, no cluster recreation.

Backtrack vs PITR: Backtrack = fast (minutes), in-place, same cluster, MySQL only, max 72h window. PITR = slower, creates new cluster, available for full retention period (35 days), both MySQL and PostgreSQL. For "accidental delete, fast recovery" → Backtrack if MySQL.
Q

A company migrates from SQL Server on-premises to AWS. Thousands of T-SQL stored procedures can't be rewritten. What Aurora feature helps?

A

Babelfish for Aurora PostgreSQL. Provides a SQL Server-compatible TDS endpoint that understands T-SQL. The app connects to Aurora as if it were SQL Server — stored procedures, drivers, and queries work without modification. You get Aurora's managed benefits while the app team migrates gradually.

Exam trigger: "SQL Server app → Aurora, minimal code change" → Babelfish. It's specifically for SQL Server to Aurora PostgreSQL migrations, not a general SQL translation tool.

Backups, Restoration, Security & Proxy

How RDS and Aurora handle data durability, how you restore from failures, how security works at every layer, and how RDS Proxy solves the Lambda connection problem.

RDS Backups
Automated backupsDaily full snapshot + transaction logs every 5 minutes → PITR to any second. Retention 1–35 days. Deleted when the RDS instance is deleted.
Manual snapshotsUser-triggered. Persist after instance deletion. No retention limit. Take before: deleting an instance, major migrations, engine upgrades.
RestoringAlways creates a new RDS instance — no in-place restore. For PITR: AWS restores closest snapshot then replays transaction logs to reach target time.
Cross-regionEnable automated backup replication to another region. Manual snapshots: copy to another region. Use for DR — restore from cross-region backup if primary region fails.
Q

A team deleted an RDS instance over the weekend to save costs. Monday they need data from it. Can they recover?

A

Only if a manual snapshot was taken before deletion. Automated backups are deleted with the instance — they don't survive instance deletion. If only automated backups existed, the data is gone. Going forward: always take a final manual snapshot before deleting any RDS instance.

Automated backup ≠ permanent storage. Manual snapshots = your permanent safety net. They persist until you explicitly delete them. This is a critical operational pattern: take manual snapshot → delete instance. Never skip the snapshot step.
Aurora Backups & Cloning
AutomatedContinuous to S3, always on, cannot be disabled. PITR to any second. No performance impact (separate storage subsystem).
BacktrackIn-place rewind, no new cluster, up to 72h window, MySQL-compatible only.
DB CloningCopy-on-write clone of entire cluster in minutes. Shares storage with source until data diverges. Much faster than backup restore. Zero production impact. Use for staging/test.
Q

You need a staging environment with a copy of production Aurora data to test a schema migration. Fastest approach with no production impact?

A

Use Aurora Database Cloning. Creates the clone in minutes using copy-on-write — it initially shares the same storage pages as production. Only when clone or production writes new data does storage diverge. Zero performance impact on production, much faster than backup restore, and cheap until data diverges significantly.

Cloning vs PITR: Cloning = fast (minutes), copy-on-write, data as of right now. PITR = slower, new cluster, data as of a specific past time. For staging/test with current data → Cloning. For recovering from a past incident → PITR.
RDS & Aurora Security
Encryption at restKMS (AES-256). Must be enabled at creation — cannot enable retroactively. To encrypt existing: snapshot → copy with encryption → restore from encrypted snapshot.
Encryption in transitTLS/SSL. Enforce via parameter group (rds.force_ssl=1 for PostgreSQL, require_secure_transport=ON for MySQL).
Network isolationAlways deploy in a private subnet. Security groups control DB port access. No public internet access unless explicitly required (avoid).
IAM DB AuthenticationUse IAM-generated auth token (valid 15 min) instead of password. Works for MySQL and PostgreSQL on RDS/Aurora. IAM role controls who can generate tokens — no stored passwords in code.
Secrets ManagerStore DB password in Secrets Manager with automatic rotation. App fetches secret at runtime via SDK. RDS integrates natively for automatic password rotation on schedule.
Audit LogsEnable DB audit logging → stream to CloudWatch Logs. Captures logins, queries, schema changes. Required for compliance.
Q

A compliance audit requires no database passwords in code or environment variables. How do you handle RDS credentials?

A

Option 1 — Secrets Manager: Store DB password as a secret. Grant the app's IAM role access. App fetches the secret at runtime via SDK. Enable automatic rotation — Secrets Manager rotates the password on schedule, app always fetches the current value. Option 2 — IAM DB Authentication: The app uses its IAM role to generate a temporary 15-min token used as the DB password. No stored password at all.

Secrets Manager = any secret, auto-rotates, all DBs. IAM Auth = zero-password, tokens only, MySQL/PostgreSQL on RDS/Aurora. Both satisfy the compliance requirement. IAM Auth is cleaner architecturally but only works for supported engines.
RDS Proxy
RDS Proxy — connection pooling between app and RDS
1000 Lambda invocations→ RDS Proxy (pool)→ RDS (20 actual connections)

Without proxy: 1000 Lambda = 1000 DB connections = DB overwhelmed. With proxy: 1000 Lambda → proxy pools to ~20 actual connections. Fully managed, serverless, auto-scales.

Problem it solvesLambda, ECS tasks open/close DB connections rapidly. DBs have a max connection limit. Too many = "too many connections" errors and connection overhead that degrades performance.
Connection poolingProxy maintains warm DB connections. App connects to proxy; proxy multiplexes onto far fewer actual DB connections. Huge reduction in connection overhead.
FailoverOn RDS/Aurora failover, proxy reconnects to new primary internally. App's connection to proxy stays stable — failover is faster and more transparent than direct connection.
IAM Auth enforcementProxy can enforce IAM auth. Apps authenticate to proxy via IAM — Secrets Manager stores the actual DB password, proxy fetches it. No DB password in app config.
VPC onlyRDS Proxy is never publicly accessible. Lambda functions that need it must be inside the same VPC (or peered VPC).
Q

Lambda functions connecting to RDS MySQL throw "too many connections" during peak load. Fix with minimum code change.

A

Add RDS Proxy in front of RDS. Update Lambda's connection string from the RDS endpoint to the Proxy endpoint — that's the only code change. The proxy pools connections: regardless of how many Lambda invocations run concurrently, the proxy maintains a small stable pool of actual DB connections. Lambda scales to thousands; DB sees dozens.

Lambda + DB = always consider RDS Proxy. Lambda is stateless and concurrent — fundamentally incompatible with traditional DB connection limits. This is the canonical exam scenario. One connection string change is all that's needed in the application.

ElastiCache

Managed in-memory data store — Redis or Memcached. Microsecond latency. Used to offload reads from databases, store session data, power leaderboards, and cache expensive computations. AWS manages provisioning, patching, failover, and backups.

The two core patterns
Pattern 1: DB Cache — lazy loading (cache-aside)
App→ check cache → ElastiCache→ cache HIT: return instantly
App→ cache MISS → nothing→ RDS→ write to cache → return to app

Cache hit = microsecond response. Cache miss = DB query + populate cache. TTL controls expiry. App manages the cache — RDS is never bypassed for writes.

Pattern 2: User Session Store — stateless app servers
User logs in → Instance 1→ write session → ElastiCache (session: user123)
Next request → Instance 2→ read session → ElastiCache→ still logged in ✓

Sessions in ElastiCache instead of instance memory = truly stateless servers. Any instance handles any user. ASG scales freely. No sticky sessions needed.

Q

RDS is hammered by repeated identical SELECT queries for a product catalogue that changes once per day. Responses are slow. Architecture fix?

A

Implement lazy loading with ElastiCache. On first request, app queries RDS and writes result to ElastiCache with 24h TTL. All subsequent requests hit ElastiCache — no RDS query. RDS load drops dramatically. Invalidate the cache key when the catalogue is updated (or let TTL expire it naturally).

Cache-aside = app manages the cache. App checks cache → falls back to DB on miss → writes to cache. DB is never bypassed for writes. TTL too short = too many misses. TTL too long = stale data. Balance against how often data actually changes.
Q

A web app stores sessions in EC2 instance memory. When ASG replaces an instance, users are logged out. Fix without sticky sessions.

A

Move session storage to ElastiCache Redis. Login writes session to Redis. Subsequent requests (any instance) read session from Redis. Instance replacements don't affect sessions — data lives in ElastiCache, not on the instance. App becomes truly stateless.

Stateless app + shared session store = canonical HA web app pattern. Eliminates need for sticky sessions (which cause uneven load). Any instance serves any user. This pattern is tested constantly — "users logged out on instance replacement" → ElastiCache sessions.

ElastiCache Redis — real-world use cases
Gaming Leaderboards

Redis Sorted Sets (ZSET). ZADD leaderboard [score] [playerId]. Top 100: ZREVRANGE 0 99. Player rank: ZREVRANK [playerId]. O(log N) ops, microsecond responses, millions of players.

Rate Limiting

Redis INCR + TTL. INCR user:123:requests → if > limit, reject. TTL auto-expires the counter. Atomic — no race conditions. Standard pattern for API throttling.

Pub/Sub Messaging

Redis Pub/Sub. Publisher sends to channel, all subscribers receive instantly. Real-time notifications, chat, live dashboards. Not durable — messages lost if no subscriber is active.

Distributed Locks

Redis SETNX (set if not exists) + TTL. Prevents multiple app instances executing the same critical section. Common for distributed job scheduling and resource coordination.

ML Feature Store

Pre-computed features stored in Redis for real-time inference. Recommendation systems and fraud detection look up features in microseconds instead of computing them on-the-fly during prediction.

Queues & Streams

Redis Lists (LPUSH/RPOP) as FIFO queues. Redis Streams for durable queues with consumer groups and acknowledgement. Alternative to SQS for teams already on Redis.

Q

A gaming company needs a real-time global leaderboard showing top 100 players and each user's rank among millions. Sub-millisecond responses required. How?

A

Use Redis Sorted Sets (ZSET) in ElastiCache. Update score: ZADD leaderboard [score] [playerId]. Top 100: ZREVRANGE leaderboard 0 99 WITHSCORES. Player rank: ZREVRANK leaderboard [playerId]. All operations O(log N) — microsecond latency at millions of players. No DB query for any read operation.

Sorted Sets are the canonical leaderboard structure. "Real-time ranking", "leaderboard", "sorted by score" → Redis Sorted Sets. A relational DB would need expensive ORDER BY + RANK window functions at scale. Redis beats it by orders of magnitude.

Exam Tips & Scenarios

The patterns that appear in Stephane's quizzes and the SAA-C03 exam. One keyword in the scenario often decides the entire answer — train yourself to spot them instantly.

RDS scenarios
Q

A reporting team runs heavy analytical queries that slow down the production RDS database. What's the right fix?

A

Create a Read Replica and point reporting queries to its endpoint. Primary handles writes unaffected. The replica can be a larger instance size to handle analytics load. The reporting team's connection string must be updated — RDS does not auto-route reads.

"Offload reads" → Read Replica. "Survive AZ failure" → Multi-AZ. The standby in Multi-AZ is NOT readable. This distinction is the most tested RDS concept on the exam.
Q

RDS in us-east-1 needs DR in eu-west-1 with RTO under 1 hour. How?

A

Create a cross-region Read Replica in eu-west-1. In a disaster, promote it to standalone (RTO = minutes). Alternatively, enable cross-region backup replication and restore from backup (simpler setup, higher RTO — depends on DB size).

RTO decides the answer. Sub-hour RTO → cross-region replica (promote it). Higher RTO acceptable → cross-region backup restore. Cross-region replica has ongoing cost; backup restore has none until disaster.
Q

Company needs OS-level access to their RDS Oracle instance to install a custom monitoring agent. What service?

A

RDS Custom for Oracle. SSH/SSM access to the underlying EC2 instance. Available for Oracle and SQL Server only.

"OS access on RDS" = RDS Custom. The triggers: "custom OS patches", "install OS agent", "access underlying OS". Standard RDS → no OS access. RDS Custom → limited OS access. EC2 self-managed → full OS access.
Aurora scenarios
Q

Need MySQL-compatible DB that survives full AZ failure with auto-failover in <30s and minimal data loss. RDS Multi-AZ takes ~60s. What do you use?

A

Aurora MySQL. 6 storage copies across 3 AZs. Promotes read replica in <30s — no data to sync (shared storage). Up to 15 read replicas any of which can be promoted. Fundamentally more resilient than RDS Multi-AZ.

"Faster failover" or "more resilient" → Aurora. RDS Multi-AZ = 60s, sync standby, 1 standby. Aurora = <30s, 6 storage copies, 15 replicas. Aurora wins on every HA metric.
Q

Developer ran bad UPDATE on Aurora PostgreSQL — all rows affected. Need fast recovery. Backtrack available?

A

Backtrack is MySQL-compatible only. For Aurora PostgreSQL, use Point-in-Time Restore — restore to a new cluster from just before the incident. Slower than Backtrack but always available. For Aurora MySQL → Backtrack (faster, same cluster). Engine matters.

Backtrack = Aurora MySQL only. If the question says PostgreSQL → PITR (new cluster). If MySQL → Backtrack option available. This nuance is tested.
ElastiCache scenarios
Q

Cache needs to persist data across restarts, support sorted leaderboards, and have automatic failover. Redis or Memcached?

A

Redis. All three requirements map to Redis features: persistence (RDB/AOF), Sorted Sets (leaderboards), Multi-AZ failover. Memcached has none of these. Whenever an exam scenario has requirements beyond "simple key-value cache" → Redis.

Memcached is correct only when: simple key-value, multi-threaded needed, data loss on restart acceptable. This is rarely the case in exam scenarios — Redis almost always wins.
Q

App behind ALB with 10 EC2 instances. Users randomly logged out when routed to different instances. Cheapest fix, minimum code change?

A

Move session storage to ElastiCache Redis. Change session handler to Redis (one config line in most frameworks). All instances share session state — any instance serves any user. Cheaper and more resilient than sticky sessions, which still fail if the pinned instance is terminated.

Sticky sessions fix the symptom, not the cause. If the sticky instance is replaced, session is lost anyway. ElastiCache fixes the root cause — session is external to any instance. Always the better architectural answer.
Q

A team says "use IAM authentication for ElastiCache Redis". Is this correct?

A

Incorrect for data operations. IAM controls the ElastiCache management API (create/delete/modify clusters via AWS API). It does NOT authenticate Redis data operations (GET, SET, ZADD etc.). For Redis data auth: use Redis AUTH (password token) or Redis RBAC (user-based). For network security: Security Groups + VPC private subnet.

This distinction is a deliberate trap. RDS supports IAM DB authentication (token-based). ElastiCache does not — Redis uses its own auth mechanism. Confusing the two is a common exam mistake.
