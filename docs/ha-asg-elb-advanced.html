Sticky Sessions

What stickiness actually does
Pins a specific user to a specific backend instance for the lifetime of their session — using a cookie as the glue.
ALB + CLB
The mechanismOn the first request, ELB picks a target normally. It then sets a cookie on the client's browser. Every subsequent request from that browser includes the cookie — ELB reads it and routes to the same instance, bypassing normal load balancing.
Duration-based cookieELB generates the cookie (AWSALB for ALB). You set the TTL (1 second to 7 days). After TTL expires, the next request picks a new instance normally. You have no control over the cookie content.
App-based cookieYour application sets its own cookie (any name except reserved AWS names). ELB reads it and sticks the user based on it. Gives you full control — you can expire stickiness when the user logs out, for example.
ScopeStickiness is configured per target group, not per listener or per ALB. Different target groups can have different stickiness settings.
With stickiness — same user always hits same instance
User (cookie: TG-A → i-001) → ALB → reads cookie → i-001 always
New User (no cookie) → ALB → round-robin → i-001 or i-002 or i-003

↑ Once new user is assigned, ALB sets AWSALB cookie → all future requests go to that same instance

The big tradeoff Stickiness breaks even load distribution. If instance i-001 gets pinned by many long-lived sessions, it bears more load than i-002 and i-003. This defeats the purpose of a load balancer. The right solution is stateless backends with a shared session store (ElastiCache Redis / DynamoDB) — use stickiness only for legacy apps you can't refactor.
Sticky sessions — scenario practice
Q

A legacy e-commerce app stores shopping cart data in the local memory of each EC2 instance. Users keep losing their carts when they're routed to different instances. What's the quickest fix, and what's the proper long-term fix?

A
Quick fix: Enable sticky sessions (duration-based) on the ALB target group. Users will be pinned to the same instance and won't lose cart state.

Proper fix: Move session/cart state to ElastiCache (Redis). All instances share the same state — now you can truly scale and route freely with no stickiness needed.
Why this matters: The exam loves "legacy app that stores state locally" as the setup. Stickiness is always presented as the quick/imperfect fix, and a shared state store (ElastiCache, DynamoDB) as the correct architectural fix. Know both.
Q

You notice that one EC2 instance behind your ALB is receiving significantly more traffic than the others, causing CPU spikes. Stickiness is enabled. What's happening and how do you fix it?

A
Stickiness is pinning a disproportionate number of long-lived sessions to one instance. New connections are distributed evenly, but sessions that started early keep returning to the same host. Fix: disable stickiness and move session state to ElastiCache, or reduce the stickiness TTL so sessions rebalance more frequently.
Why this matters: "Uneven load distribution + stickiness enabled" is a direct exam pattern. The answer is always that stickiness is the cause, and the architectural fix is stateless backends.
§2

Cross-Zone Load Balancing

The problem it solves
When AZs have unequal numbers of instances, without cross-zone LB some instances get far more traffic than others.
Without cross-zone — traffic is uneven
DNS routes 50% → AZ-A node → AZ-A has 2 instances → 25% each
DNS routes 50% → AZ-B node → AZ-B has 8 instances → 6.25% each

↑ AZ-A instances each handle 4× more traffic than AZ-B instances. Hot AZ.

With cross-zone — traffic is even across all 10 instances
50% to AZ-A node → Distributed across all 10 instances → 10% each
50% to AZ-B node → Same pool → 10% each
ALBCross-zone LB is always on, cannot be disabled, no charge for cross-AZ data transfer.
NLB / GWLBCross-zone LB is off by default. Can be enabled per load balancer. Data transfer charges apply when traffic crosses AZ boundaries.
Classic LBOff by default. Free when enabled (no inter-AZ data transfer charge). Legacy — don't use for new workloads.
Cross-zone — scenario practice
Q

You have an NLB with 2 instances in AZ-A and 8 instances in AZ-B. Cross-zone load balancing is disabled. A performance test shows AZ-A instances are at 80% CPU while AZ-B instances are at 20%. What do you do?

A
Enable cross-zone load balancing on the NLB. Traffic will distribute evenly across all 10 instances (10% each) instead of 50% landing on 2 AZ-A instances. Note: this will incur inter-AZ data transfer charges on NLB — evaluate the cost tradeoff.
Why this matters: Uneven CPU across AZs when instance counts differ = cross-zone is disabled. The fix is enabling it. On NLB/GWLB remember to mention the data transfer cost — it shows the examiner you know the nuance.
Q

Your team is seeing unexpected inter-AZ data transfer charges on an NLB that just had cross-zone load balancing enabled. Is this expected?

A
Yes, completely expected. Unlike ALB (where cross-zone is free), NLB and GWLB charge for cross-AZ data transfer when cross-zone LB is enabled. The charge is the standard inter-AZ data transfer rate. To avoid it: keep cross-zone off and ensure equal instance counts across AZs instead.
Key fact to memorise: ALB cross-zone = always on, always free. NLB/GWLB cross-zone = off by default, charged when on.

SSL & SNI

How TLS termination works at ELB
ELB terminates the HTTPS connection from the client, decrypts it, then makes a new plain HTTP (or HTTPS) connection to the backend. Certs live on the ELB, not the instances.
ALB + NLB
ACM integrationUse AWS Certificate Manager to provision and auto-renew SSL certs for free. Just associate the ACM cert ARN with your HTTPS listener. ACM handles renewal — no more expired cert incidents.
Security policyELB security policies define which TLS versions (TLS 1.0, 1.1, 1.2, 1.3) and cipher suites are allowed. Use ELBSecurityPolicy-TLS13-1-2-2021-06 or newer in production to enforce TLS 1.2+ and strong ciphers.
HTTPS → HTTP backendStandard pattern. ELB terminates TLS, sends plain HTTP to backend. Backend doesn't need to manage certs. Works because traffic from ELB to backend stays within the VPC (private network).
End-to-end encryptionIf compliance requires encryption all the way to the instance: ALB → HTTPS listener → backend target with self-signed cert. Or use NLB with TLS passthrough — NLB passes encrypted traffic without decrypting.
SNI — Server Name Indication
Allows multiple SSL certificates on a single ALB listener — one per hostname. The client announces which hostname it wants in the TLS handshake, and ALB picks the right cert.
ALB + NLB only
SNI — one ALB, three domains, three certs
Client → app.example.com → TLS Hello (SNI: app.example.com) → ALB serves *.example.com cert → app TG
Client → api.example.com → TLS Hello (SNI: api.example.com) → ALB serves api cert → api TG
Client → tenant1.saas.io → TLS Hello (SNI: tenant1.saas.io) → ALB serves tenant1 cert → tenant1 TG

↑ All three share the same :443 listener. No separate ALBs needed. Classic LB does NOT support SNI.

SupportsALB and NLB. Classic Load Balancer does not support SNI — one cert per CLB.
Max certs per ALB25 certificates per HTTPS listener (can be increased via limit request). Enough for most multi-tenant SaaS.
Wildcard certsA single *.example.com wildcard cert covers all subdomains — use instead of individual certs where possible.
SSL / SNI — scenario practice
Q

You're building a SaaS platform where each customer gets their own subdomain (customer1.app.com, customer2.app.com) each needing its own SSL certificate. What's the most cost-effective and operationally simple way to handle this on AWS?

A
Use a single ALB with SNI. Add each customer's ACM certificate to the HTTPS :443 listener. ALB uses SNI to serve the correct cert per subdomain. Use host-based routing rules to direct each subdomain to the correct target group. No need for one ALB per customer — that would be expensive and a management nightmare.
Why this matters: Multi-tenant SSL on one ALB is a very common architecture question. SNI is the answer. The trap is suggesting Classic LB — it doesn't support SNI so you'd need a separate CLB per domain.
Q

A financial services company requires that all traffic be encrypted in transit, including between the load balancer and backend EC2 instances. How do you architect this?

A
Two options:

Option 1 — NLB with TLS passthrough: NLB forwards encrypted traffic directly to instances without decrypting. Instances terminate TLS themselves. Full end-to-end encryption, NLB never sees plaintext.

Option 2 — ALB with HTTPS to backend: ALB terminates TLS, then re-encrypts to the backend. Backend uses a self-signed or ACM private CA cert. ALB does see plaintext (between termination and re-encryption).
When to use which: If "NLB never sees plaintext" is a hard requirement → Option 1. If you just need encryption in transit (compliance checkbox) → Option 2 is simpler operationally.
§4

Connection Draining

Deregistration Delay (ALB/NLB) = Connection Draining (CLB)
When a target is being removed, ELB lets in-flight requests finish before completely cutting off the instance — instead of killing active connections mid-request.
All ELB types
What happens during deregistration
Instance marked for removal (or unhealthy)
↓
ELB stops sending NEW requests to this instance
↓ (deregistration delay timer starts — default 300s)
In-flight requests continue until they complete
↓ (timer expires OR all connections close, whichever first)
Instance fully deregistered — connection forcibly closed
Default value300 seconds (5 minutes). Can be set from 0 to 3600 seconds.
Set to 0Disables connection draining entirely — connections are immediately closed on deregistration. Use only if your requests are very short-lived (e.g. <1s) and you want fast instance replacement.
Set low (e.g. 30s)For microservices with short-lived HTTP requests. Fast deploys, faster ASG scale-in.
Set high (e.g. 600s+)For long-running requests — file uploads, long-polling, report generation. Give them time to complete before the instance disappears.
Connection draining — scenario practice
Q

Users are reporting that during deployments (when old instances are replaced), they occasionally get errors mid-request — as if the connection was just cut. What ELB setting do you check?

A
Check the deregistration delay (connection draining). It may be set to 0 (disabled) or set too low relative to how long your requests take. Increase it to a value comfortably above your p99 request duration. Requests that started before the drain window closes will now complete cleanly.
Exam pattern: "Errors during deployments / instance replacement" = deregistration delay too low. "Deployments are slow / ASG scale-in takes too long" = deregistration delay too high.
Q

Your ASG is scaling in (removing instances) but it's taking much longer than expected, causing you to pay for extra instances during off-peak hours. What setting should you lower?

A
Lower the deregistration delay. The default is 300s — if your requests are short (API calls typically complete in <1s), the instance sits idle for up to 5 minutes waiting for connections that have already closed. Set it to 30s or even lower for fast-terminating services.
Cost implication: High deregistration delay = slow scale-in = paying for idle instances. This is a direct cost-optimisation pattern.
§5

Auto Scaling Groups

What an ASG does
Maintains a fleet of EC2 instances at the right size automatically — scaling out when demand rises, scaling in when it drops, and replacing unhealthy instances without your intervention.
Min / Desired / Max Min: Never go below this count (floor — for baseline availability).
Desired: ASG tries to maintain this count right now. Can be changed manually or by scaling policies.
Max: Never go above this count (ceiling — for cost control).
Launch TemplateDefines what each instance looks like: AMI, instance type, key pair, security groups, IAM role, user data, EBS volumes. Think of it as the blueprint. Use Launch Templates (not the older Launch Configurations — LCs are immutable and lack features).
ELB integrationASG registers new instances with the ALB/NLB target group automatically. When an instance is terminated, ASG deregisters it (respecting the deregistration delay). This is the core HA + scaling pattern.
Health checksBy default, ASG uses EC2 health checks (is the instance running?). Enable ELB health checks on the ASG too — this means if ELB marks an instance unhealthy (app is failing, not just the instance), ASG will terminate and replace it. Critical for production.
Multi-AZAlways span your ASG across at least 2 AZs (ideally 3). ASG tries to keep instance count balanced across AZs. If an AZ goes down, ASG launches replacements in the remaining AZs.
Free serviceASG itself costs nothing. You only pay for the EC2 instances it launches.
§6

Scaling Policies

The four scaling policy types
1. Target Tracking Scaling
You set a target metric value. ASG automatically adds/removes instances to keep the metric at that target. The simplest and most recommended policy.
Recommended
How it worksPick a metric and a target value. ASG does the math. e.g. "keep average CPU at 50%" — if CPU goes to 70%, ASG adds instances. If CPU drops to 20%, ASG removes instances. You never write scale-out or scale-in rules separately.
Common targets
  • ASGAverageCPUUtilization — most common, target 40–60%
  • ALBRequestCountPerTarget — keep requests per instance at N
  • ASGAverageNetworkIn/Out — for network-bound workloads
Scale-in protectionTarget tracking creates both scale-out and scale-in alarms automatically. To prevent aggressive scale-in, set a scale-in cooldown or use instance scale-in protection on specific instances.
2. Simple / Step Scaling
CloudWatch alarm triggers a specific scaling action — add N instances when alarm fires, remove N when alarm clears. Step adds more granularity by defining different actions at different alarm thresholds.
More control
Simple ScalingOne alarm → one action. Add 2 instances when CPU > 70%. Remove 1 instance when CPU < 30%. Waits for cooldown before triggering again — can be slow to react to sudden spikes.
Step ScalingMultiple thresholds, different actions per threshold. e.g.: CPU 70–80% → add 1 instance. CPU 80–90% → add 2. CPU >90% → add 4. Reacts proportionally to severity. Does not wait for cooldown — can continuously add instances during a spike.
When to useWhen target tracking doesn't give you enough control — e.g. you want to scale aggressively during sudden traffic spikes (step) or when you have a custom CloudWatch metric as the trigger.
3. Scheduled Scaling
Scale at a known time — useful when you know traffic patterns in advance. Cron expression or one-time scheduled action.
Predictable patterns
How it worksSet a future time (or recurring cron) and a desired capacity. ASG adjusts min/desired/max at that exact time. e.g. "every weekday at 8am set desired to 10, at 8pm set desired to 2".
Classic use casesBusiness-hours traffic patterns, known events (Monday morning spike, end-of-month payroll processing, Black Friday). Scale up before the traffic arrives — don't wait for CloudWatch to react after the fact.
Combine with reactiveScheduled scaling sets the baseline. Target tracking reacts to unexpected variation on top. Use both together.
4. Predictive Scaling
ML-based. AWS analyses historical CloudWatch data and forecasts future load — then proactively schedules scaling actions ahead of time.
ML-powered
How it worksAWS looks at the last 14 days of metric history, detects recurring patterns, and creates scheduled scaling actions automatically. You don't manually define the schedule — AWS predicts it.
Best forWorkloads with repeating patterns: weekly cycles, daily office hours, monthly batch peaks. Needs at least 24 hours of historical data to generate forecasts.
vs ScheduledScheduled = you know the pattern and set it manually. Predictive = AWS detects the pattern automatically from your history. Use predictive when the pattern is regular but you don't want to maintain cron expressions.
Critical ASG mechanics
Cooldown periodAfter a scaling activity, ASG waits N seconds before evaluating another scaling trigger (default 300s). Prevents "flapping" — rapidly adding then removing instances before the first batch has had time to affect the metric. During cooldown, alarms that fire are ignored.
Warm-up periodNew instances take time to be ready (boot, app init, cache warm). During warm-up, ASG doesn't count the new instance's metrics toward the aggregate. Prevents premature scale-out thinking the new instance "didn't help" when it's still booting.
Termination policyWhen scaling in, which instance does ASG kill? Default: oldest launch template → then the instance closest to the next billing hour. Can customise: OldestInstance, NewestInstance, ClosestToNextInstanceHour, Default. For blue/green deploys, OldestLaunchTemplate is common.
Instance refreshRolling replacement of all instances in the ASG with a new launch template version. Specify a minimum healthy percentage (e.g. 90%) — ASG keeps 90% healthy while replacing the rest. No downtime. The built-in way to do rolling deploys on an ASG.
Lifecycle hooksPause an instance in a transition state (Pending:Wait or Terminating:Wait) and run custom logic — e.g. pull config from S3, drain a queue, deregister from Consul, run a cleanup script. Instance is held until you send CompleteLifecycleAction or the timeout fires.
Scale-in protectionMark specific instances as protected from scale-in. ASG will never terminate them automatically. Use for instances holding critical state or running a long job you don't want interrupted.
ASG + Scaling Policies — scenario practice
Q

Your e-commerce site gets a massive traffic spike every day at 9am when your marketing email goes out. By the time CloudWatch detects high CPU and ASG scales out, it's already been 5 minutes and users experienced slow load times. How do you fix this?

A
Use Scheduled Scaling to pre-scale at 8:50am — before the spike hits. Set desired capacity to your expected peak count 10 minutes before the email sends. Combine with Target Tracking to handle unexpected variation beyond the scheduled baseline.
The principle: Reactive scaling (target tracking, step) always lags — it needs the problem to occur before it can react. When you know the spike is coming, scheduled scaling is always faster. Pre-scale beats react-and-catch-up.
Q

During a sudden traffic spike, your step scaling policy fired and added 4 instances. But 2 minutes later it fired again and added 4 more — and the CPU was actually coming down. Now you have too many instances. What caused this and how do you prevent it?

A
The cooldown period is too short (or the warm-up period isn't set). The first 4 instances hadn't finished booting and accepting traffic yet, so the metric hadn't improved. ASG saw the alarm still firing and added 4 more. Fix: increase the cooldown period to give new instances time to become healthy and affect the metric, and/or set an instance warm-up period in the scaling policy.
Exam pattern: "ASG keeps adding instances even though the problem is resolved / over-scales" = cooldown too short or warm-up not set. "ASG is slow to scale out" = cooldown too long.
Q

You're deploying a new version of your app. You want all existing instances replaced with the new AMI, but you can't have more than 20% of instances down at any point. What's the cleanest way to do this?

A
Use Instance Refresh on the ASG with a minimum healthy percentage of 80%. ASG will update the launch template, then perform a rolling replacement — terminating at most 20% of instances at a time, waiting for replacements to pass health checks before continuing to the next batch.
vs other options: Blue/green (create new ASG) is heavier. Manual termination is risky. Instance Refresh is the built-in, zero-downtime rolling deploy mechanism for ASGs.
Q

When ASG scales in and terminates an instance, a long-running background job that was running on it gets killed mid-execution, corrupting data. How do you prevent this?

A
Use an ASG Lifecycle Hook on the EC2_INSTANCE_TERMINATING transition. The hook pauses the instance in Terminating:Wait state. Your termination script (triggered by EventBridge or SSM) detects the lifecycle event, waits for the job to finish (or saves checkpoint state to S3/DynamoDB), then calls CompleteLifecycleAction to let ASG proceed with termination.
Lifecycle hooks in one sentence: They give you a window of time to run custom logic before an instance fully launches or terminates — the instance is paused in a wait state until you signal it's OK to proceed.
Q

Which scaling policy would you use for a workload that follows repeating weekly patterns (high on weekdays, low on weekends) but you don't want to manually maintain the schedule?

A
Predictive Scaling. It analyses your CloudWatch metric history, detects the weekday/weekend pattern automatically, and creates proactive scaling schedules without you maintaining cron expressions. Needs 24+ hours of data and works best with at least 14 days of history.
Predictive vs Scheduled: Scheduled = you define the times. Predictive = AWS learns the times from your history. If the exam says "automated", "no manual schedule", "repeating pattern" → Predictive Scaling.
