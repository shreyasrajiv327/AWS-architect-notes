EC2 — Elastic Compute Cloud

Virtual servers in the cloud. The foundation of AWS compute. You pick the OS, CPU, RAM, storage, and network. Everything else on this page (ENI, EBS, EFS) attaches to EC2 instances.

Instance families — pick the right shape
General Purpose T3 · M6i · M7g

Balanced CPU/RAM/network. T-series are burstable — earn CPU credits when idle, spend on spikes. Great for web servers, dev/test, small DBs. M-series = steady, no bursting.

Compute Optimized C6i · C7g

High CPU-to-RAM ratio. Use for batch processing, media transcoding, HPC, ML inference, gaming servers. Choose when your workload is CPU-bound and memory is secondary.

Memory Optimized R6i · X2idn

High RAM-to-CPU. SAP HANA, in-memory caches (Redis at scale), real-time big data analytics. X-series can reach 24 TB RAM.

Storage Optimized I4i · D3

High sequential I/O with local NVMe SSDs (instance store). NoSQL DBs, Hadoop HDFS, data warehouses. Instance store is ephemeral — not for persistent data.

Accelerated Computing P4 · G5 · Inf2

GPUs and custom AWS chips (Trainium/Inferentia). ML training, deep learning, HPC, graphics. Inf2 is purpose-built for low-cost ML inference.

Purchasing options — always an exam topic
On-Demand

Pay per second (Linux) or per hour. No commitment, most expensive. Use for unpredictable workloads, short-term tests, or traffic you can't forecast.

Reserved Instances

1 or 3-year commit → up to 72% off On-Demand. Standard RI = locked to family/region. Convertible RI = can change family, OS, tenancy. Use for steady-state prod.

Savings Plans

Commit to $/hr spend (not instance type) for 1–3 years. Compute SP covers EC2 + Fargate + Lambda. More flexible than RIs, similar discounts. The modern recommended path.

Spot Instances

Up to 90% off — unused EC2 capacity. AWS can reclaim with 2-min warning. Use for fault-tolerant, stateless, interruptible work: batch jobs, ML training, CI workers.

Dedicated Hosts

Physical server reserved for you. Use for compliance, BYOL (Oracle, Windows Server), or regulatory requirements. Most expensive option.

Key concepts
AMIAmazon Machine Image — OS + software snapshot used to launch instances. Region-specific, can be copied cross-region. Public (Amazon/community), private (yours), or Marketplace.
User DataBootstrap script run once on first boot as root. Used to install packages, pull config, start services. Max 16 KB.
Instance MetadataHTTP endpoint at 169.254.169.254/latest/meta-data/. Query AMI-ID, instance type, IAM role creds, public IP. IMDSv2 (session-based tokens) is the secure default.
Security GroupsStateful virtual firewall at the ENI level. Only allow rules — no deny. Changes apply immediately. Default: deny all inbound, allow all outbound.
Placement GroupsCluster — same rack, ultra-low latency, HPC. Spread — max 7 instances/AZ, different hardware, HA. Partition — isolated hardware groups, HDFS/Cassandra.
TenancyShared (default) · Dedicated Instance (no cross-account hardware sharing) · Dedicated Host (full physical server control for BYOL/compliance).
HibernateRAM state saved to root EBS volume. Resume from exact memory state. Root EBS must be encrypted. Max 60 days. Great for long-initialization services.
Nitro SystemAWS hypervisor replacing Xen. Nearly all modern instances. Enables higher networking (100 Gbps), EBS bandwidth, EFA, and bare-metal instances. Required for io2 Multi-Attach.
Production pattern — Auto Scaling Group Always run EC2 in an ASG: define min/desired/max. Attach to ALB/NLB. Use lifecycle hooks for graceful connection drain. Set termination policies. Combine On-Demand base capacity with Spot for burst savings (mixed instances policy).
Cost tip — Graviton (ARM) M7g/C7g/R7g (Graviton 3) offer 20–40% better price-performance vs x86 equivalents. Use for new workloads where ARM compatibility is confirmed.
T-series gotcha T3/T4g instances earn CPU credits when idle and spend them on spikes. If consistently at high CPU, credits exhaust and performance drops to the baseline (~10–30% vCPU). Fix: switch to M-series, or enable T Unlimited mode (billed for extra credits).
EC2 — scenario practice
Q

A company runs a stateless image-processing job that takes 2 hours to complete. It can restart from a checkpoint if interrupted. They want the lowest possible cost. Which purchasing option?

A

Spot Instances. The job is stateless, fault-tolerant, and restartable — exactly what Spot is designed for. Up to 90% cheaper than On-Demand. Use a Spot interruption handler to save the checkpoint before the 2-minute reclaim window closes.

The pattern: "Can restart", "checkpoint", "fault-tolerant", "batch", "stateless" → Spot. If any of those words are missing and the job cannot be interrupted → On-Demand or Reserved.
Q

A production database runs steadily 24/7 for the next 3 years. The team wants maximum cost savings. Reserved Instance or Savings Plan — which and why?

A

Standard Reserved Instance (3yr, all upfront) gives up to 72% discount — slightly higher than Compute Savings Plans (~66%). Since the workload is pure EC2, steady-state, and the instance family won't change, the RI lock-in is acceptable. Savings Plans win when you also run Fargate or Lambda.

Decision rule: EC2-only + stable instance type → Standard RI. Mixed compute (EC2 + Fargate + Lambda) or need flexibility to change instance types → Compute Savings Plan.
Q

An application on a t3.medium suddenly shows degraded performance during business hours even though no code changed. CPU metrics show consistent 95%+ usage. What's the root cause and fix?

A

The t3.medium has exhausted its CPU credits. T-series instances run at a baseline CPU % and accumulate credits when idle. When credits run out under sustained load, the instance is throttled to its baseline (~20% for t3.medium). Fix: enable T Unlimited mode to burst beyond credits (you pay for the extra), or migrate to an M-series (M6i/M7g) which has no burst concept — just steady full CPU.

Exam tell: "Performance degraded gradually" or "worked fine at first then got slow" on a T-series instance = CPU credit exhaustion. "Sudden" degradation at a specific time = something else (traffic spike, code change).
Q

A financial firm needs to run Oracle Database with BYOL (Bring Your Own License). Oracle licenses are tied to physical CPU cores. Which EC2 tenancy model is required?

A

Dedicated Host. Oracle BYOL requires you to control the physical server to count and manage CPU sockets/cores for licensing. Dedicated Hosts give you a full physical server and visibility into its sockets, cores, and hardware. Dedicated Instances give isolation but no hardware visibility — not enough for Oracle BYOL.

Dedicated Host vs Dedicated Instance: Both isolate you from other AWS customers. Only Dedicated Hosts give you visibility into physical hardware details — needed for BYOL licensing that counts physical cores (Oracle, Windows Server).

ENI — Elastic Network Interface

A virtual NIC that lives in a VPC subnet. Every EC2 instance has at least one (eth0 = primary ENI). You can attach multiple ENIs, move them between instances, and configure networking at the ENI level independently.

What an ENI contains
Private IPv4 address(es)

One primary private IP (from subnet CIDR) + optional secondary IPs. Secondary IPs let one instance host multiple websites/services. Primary IP persists if ENI is detached.

Elastic IP (optional)

Static public IPv4 attached to a private IP on the ENI. Survives stop/start cycles. One EIP per private IP. Charged when allocated but not in use — always associate promptly.

MAC Address

Fixed per ENI. Useful for software licensing tied to MAC address. Move the ENI → keep the MAC → license survives instance replacement.

Security Groups

SGs attach to ENIs, not instances. An instance with 2 ENIs can have different SGs on each. Swap SGs on a live ENI instantly — no restart needed.

Source/Dest Check

Default ON — EC2 drops packets where it isn't src or dst. Must be disabled for NAT instances, VPN gateways, or any instance routing traffic for others.

IPv6 addresses

Optional. All AWS IPv6 are globally routable by default. For outbound-only IPv6 (no inbound), use an Egress-Only Internet Gateway — the IPv6 equivalent of NAT.

ENI types
Standard ENIDefault. Up to 15 per instance (large types). Suitable for all general networking. Number of allowed ENIs and secondary IPs scales with instance size.
ENA (Elastic Network Adapter)Enhanced networking driver — up to 100 Gbps, low latency, high packets-per-second. Standard on all modern instance types. Replaces older ixgbevf SR-IOV driver.
EFA (Elastic Fabric Adapter)OS-bypass networking for HPC and ML. Used with MPI workloads and NCCL for multi-GPU distributed training. Not for general use — specific instance types only.
Key behaviors and use cases
Hot attach/detachAttach or detach ENIs from running instances (secondary ENIs only — primary eth0 cannot be detached). Requires OS-level driver support.
Network failoverMove a secondary ENI (with its private IP and EIP) from a failed instance to a standby instance in seconds. No DNS change needed — the IP travels with the ENI.
Dual-homed instanceTwo ENIs in different subnets (e.g., public + private) gives the instance a presence in both. Classic use: firewalls, proxies, bastion hosts needing strict network isolation.
ENI limitsMax ENIs per instance scales with instance size. Example: t3.micro = 2 ENIs · c5.4xlarge = 8 ENIs · p3.16xlarge = 8 ENIs. Each ENI also has a max secondary IPs limit.
VPC flow logsCaptured at the ENI level. You can enable flow logs per-ENI, per-subnet, or per-VPC. Useful for debugging connectivity and security auditing.
Exam scenario — ENI failover Highly available database on EC2, minimal failover disruption required. Answer: move the secondary ENI (with its EIP) to a standby instance. DNS-based failover has TTL delays (60s+). ENI move is near-instant and the IP doesn't change.
Production pattern — multi-ENI segmentation Management traffic on eth0 (private subnet, admin SG). Application traffic on eth1 (public subnet, app SG). Eliminates risk of accidentally exposing management endpoints through app SG misconfiguration.
Source/dest check If an EC2 instance is acting as a NAT instance or VPN router and traffic isn't flowing through it, the answer is almost always: disable source/destination check on its ENI. This check is ON by default and prevents packet forwarding.
ENI — scenario practice
Q

A highly available MySQL database runs on EC2 with a standby instance. The primary fails. You need to recover with the same IP address and minimal disruption. DNS failover takes 60+ seconds due to TTL. What's the fastest solution?

A

Detach the secondary ENI (with its Elastic IP) from the failed primary and attach it to the standby instance. The IP address doesn't change — clients reconnect to the same IP immediately. No DNS TTL wait, no code change, no client reconfiguration. Recovery time is seconds, not minutes.

Why not DNS failover: Route 53 failover is limited by TTL — even with a 60s TTL, clients may cache longer. ENI move is layer 3 — the IP is live on the new instance almost instantly.
Q

You set up an EC2 instance as a NAT instance so private subnet instances can reach the internet. Private instances can't get outbound connectivity. The instance is running and has the right route table entries. What are you missing?

A

Disable the source/destination check on the NAT instance's ENI. By default, EC2 drops any packet where it isn't the source or destination. A NAT instance forwards other instances' packets — those packets have the private instance as the source, not the NAT instance — so they get silently dropped. Disabling src/dst check allows forwarding.

This is the #1 NAT instance gotcha. Everything else can be correct (routes, SGs, IGW) and it still won't work until src/dst check is disabled. Same applies to VPN appliances or any EC2 acting as a router.
Q

A software vendor's license is locked to a specific MAC address. Your team terminates the old EC2 instance and launches a new one. The license stops working. How do you fix this for future replacements?

A

Attach a dedicated ENI to the instance that holds the licensed MAC address. When you need to replace the instance, detach the ENI from the old one and attach it to the new one. The MAC address travels with the ENI — the license remains valid across any number of instance replacements.

MAC address is per-ENI, not per-instance. This is a real production pattern used with Windows Server, Oracle, and other MAC-locked licensing schemes.

EBS — Elastic Block Store

Network-attached persistent block storage. Behaves like a hard drive plugged into EC2. Data persists independently of instance lifecycle. AZ-scoped — a volume in us-east-1a cannot attach to an instance in us-east-1b.

Volume types — know these cold
gp3 — General Purpose SSD Default

Baseline 3,000 IOPS + 125 MB/s regardless of size. Provision up to 16,000 IOPS and 1,000 MB/s independently. Use for most workloads: boot volumes, web servers, small-medium DBs. Cheaper than gp2 for equivalent performance.

io2 Block Express — Provisioned IOPS SSD Perf

Up to 256,000 IOPS + 4,000 MB/s. Multi-Attach capable (up to 16 Nitro instances, same AZ). 99.999% durability. Use for Oracle, SQL Server, mission-critical OLTP. io1 is the older, less capable predecessor.

st1 — Throughput Optimized HDD Sequential

Max 500 MB/s, low cost per GB. Large sequential reads. Cannot be a boot volume. Use for data warehouses, log processing, MapReduce, Kafka. Poor for random I/O.

sc1 — Cold HDD Archive

Max 250 MB/s. Lowest-cost EBS option. Infrequently accessed sequential data. Cold data lakes, backups, archives. Cannot be a boot volume.

Key features
SnapshotsPoint-in-time incremental backups stored in S3 (managed by AWS). Only changed blocks after first snapshot. Use to clone volumes, copy across AZs/regions, share. Volume remains available during snapshot creation.
EncryptionAES-256 via AWS KMS. Covers data at rest, in transit between EC2↔EBS, and all snapshots/copies. Set "Always encrypt new EBS volumes" as an account-level default. Unencrypted → encrypted: copy snapshot with encryption enabled, then create volume.
Multi-Attachio1/io2 only. One volume → up to 16 Nitro instances in the same AZ. The application must manage concurrent writes (use cluster-aware filesystems like GFS2). Use for Oracle RAC, high-availability stateful apps.
EBS-OptimizedDedicated network bandwidth for EBS I/O, separate from general instance network traffic. Enabled by default on most modern instances. Critical for IOPS-heavy workloads to avoid I/O starvation.
Elastic VolumesModify type, size, and IOPS on a live volume without downtime. Increase only — cannot shrink. After expanding, run growpart + resize2fs (ext4) or xfs_growfs (XFS) in the OS to use the new space.
Fast Snapshot RestoreEliminates first-access I/O latency (initialization). Pre-warms volumes from snapshots. Extra cost per AZ. Use when you frequently provision from snapshots at scale (AMI-based fleets).
Delete on TerminationRoot volume default: ON. Data volumes default: OFF. A common prod mistake is losing root volumes on termination. Set explicitly in launch templates.
AZ boundary — the #1 gotcha EBS volumes are locked to one AZ. To move: snapshot → create new volume in target AZ from snapshot. For cross-region: copy the snapshot to the other region first, then create volume.
Production pattern — automated snapshot lifecycle Use AWS Backup or Data Lifecycle Manager (DLM) to automate daily snapshots, set retention windows, and copy snapshots cross-region for DR. Never rely on manual snapshots for production databases.
gp2 vs gp3 — know the difference gp2 IOPS = 3 × size in GB (min 100, max 16,000). You need 5,333 GB to hit 16K IOPS on gp2. gp3 IOPS are provisioned independently — 16,000 IOPS on a 1 GB volume. For new workloads, always prefer gp3. Exam trap: "need more IOPS → increase volume size" is gp2 logic.
IOPS vs Throughput — which metric matters? IOPS = random small I/O ops per second. Throughput = MB/s for sequential large blocks. Databases → IOPS (io2/gp3). Data warehouses, Kafka, log processing → Throughput (st1). The exam will describe a workload — pick the right metric, then the right volume type.
EBS — scenario practice
Q

A team needs to move an EBS volume from us-east-1a to us-east-1b because their new EC2 instance was launched in the wrong AZ. How?

A

EBS volumes are AZ-locked — you cannot directly move them. The two-step process: (1) Create a snapshot of the volume in us-east-1a. (2) Create a new volume from that snapshot, selecting us-east-1b as the AZ. Attach the new volume to the instance. For cross-region moves, add a step: copy the snapshot to the target region first.

Always a 2-step for same-region, 3-step for cross-region. Snapshot → new volume (same region) or snapshot → copy snapshot → new volume (cross-region). This exact flow appears on the exam regularly.
Q

A database on gp2 needs more IOPS. A colleague says "just increase the volume size to get more IOPS." You're using gp3. Is the colleague right?

A

Wrong for gp3. On gp2, IOPS are tied to size (3 IOPS/GB) so increasing size increases IOPS — your colleague is thinking of gp2. On gp3, IOPS are provisioned independently of size. To get more IOPS on gp3, just increase the provisioned IOPS value directly — no need to pay for extra storage you don't need.

This is the most common gp2/gp3 trap. If you're on gp2 and need 16,000 IOPS, you need 5,333 GB. On gp3, you set 16,000 IOPS on a 1 GB volume. Always migrate to gp3 for new workloads.
Q

A security audit requires that all EBS volumes be encrypted, but there's an existing unencrypted production volume with data on it. How do you encrypt it without losing the data?

A

You cannot encrypt an EBS volume in-place. The 4-step process: (1) Snapshot the unencrypted volume. (2) Copy the snapshot with encryption enabled (KMS key selected during copy). (3) Create a new encrypted volume from the encrypted snapshot. (4) Detach the old volume and attach the new encrypted one. The data is preserved throughout.

No direct encryption toggle exists. The snapshot copy step is the only way. Going forward, enable "Encrypt new EBS volumes by default" at the account level so new volumes are always encrypted.
Q

An application writes large sequential log files to EBS. It's experiencing throughput bottlenecks but IOPS usage is low. Which volume type should you migrate to?

A

st1 (Throughput Optimized HDD). Low IOPS + high sequential write = a throughput-bound workload, not IOPS-bound. st1 is optimised for exactly this: large sequential reads/writes at up to 500 MB/s at a low cost per GB. SSD (gp3/io2) would be wasted money — you don't need random I/O performance here.

The mental model: If the workload reads/writes large files from start to end (logs, video, Kafka topics, MapReduce) → HDD (st1). If it reads/writes random small blocks (databases, OS, app servers) → SSD (gp3/io2).

EFS — Elastic File System

Fully managed, serverless NFS file system. Scales automatically from zero to petabytes. Can be mounted concurrently by thousands of EC2 instances across multiple AZs — no capacity planning required.

EFS vs EBS — the core distinction
ProtocolNFSv4.1/v4.0 — mounts as a shared network filesystem. EBS = block device (xvda, nvme0n1). EFS = POSIX filesystem path (/mnt/efs).
ConcurrencyThousands of instances across multiple AZs mount simultaneously with full read/write access. EBS: one instance only (except io2 Multi-Attach in same AZ, max 16).
ScaleFully elastic — grows and shrinks automatically. No capacity planning. EBS: you provision a fixed size, then expand manually via Elastic Volumes.
OS compatibilityLinux only (NFS). Windows instances cannot mount EFS natively. For Windows shared storage → Amazon FSx for Windows File Server (SMB/NTFS).
Cost modelPay per GB stored + retrieval fees on IA tiers. No provisioned capacity. More expensive per GB than EBS. Use storage classes aggressively to control cost.
Use casesShared content repos, CMS media libraries, home directories, container persistent storage (ECS/EKS), ML training datasets, lift-and-shift NFS workloads.
Storage classes and lifecycle
EFS Standard

Multi-AZ redundancy. Highest cost, best durability and availability. Use for frequently accessed production files requiring 99.99% SLA.

EFS Standard-IA

Same multi-AZ durability, lower storage cost, per-access retrieval fee. Enable Intelligent-Tiering to auto-move files to IA after N days without access.

EFS One Zone

Single AZ. Up to 47% cheaper than Standard. Suitable for dev/test, backups, or workloads where AZ resilience is handled at the application layer.

One Zone-IA

Cheapest EFS tier. Single AZ + per-access retrieval. Best for dev environments or infrequently accessed data that doesn't need multi-AZ durability.

Throughput modes
Elastic ThroughputAutomatically scales up to 3 GB/s read / 1 GB/s write. No baseline. Recommended for most new workloads — handles spiky or unpredictable access patterns.
Provisioned ThroughputFixed MB/s regardless of storage size. Use when throughput needs consistently exceed what Elastic mode provides at your storage size. Higher cost.
Bursting ThroughputLegacy. Throughput tied to file system size (50 MB/s per TB stored, burst to 100 MB/s per TB). Burst credits accumulate when idle. Can throttle small, heavily accessed file systems.
Security and access
Mount targetsOne mount target per AZ (created in a subnet). Instances connect to the mount target in their AZ. Each mount target has an IP + Security Group. You need one per AZ you want to access EFS from.
Security GroupsMount target SG must allow NFS inbound (port 2049) from instance SGs. Instance SG must allow NFS outbound (the EFS mount helper handles this automatically).
EncryptionAt-rest: KMS-managed, set at creation (cannot enable retroactively). In-transit: TLS via amazon-efs-utils mount helper. Always enable both in production.
EFS Access PointsEnforce a POSIX user/group identity and a root directory path per client. Combine with IAM to give containers or Lambda isolated views into the same EFS filesystem — no cross-contamination of data.
VPC-native onlyEFS is VPC-bound. On-premises access requires Direct Connect or VPN. Cross-region replication → use AWS DataSync.
Production pattern — EFS with ECS/EKS In ECS: define EFS volumes in task definitions with mount points. In EKS: use the EFS CSI driver with PersistentVolumeClaims set to ReadWriteMany access mode. EFS is the only AWS-native storage that supports RWX across multiple nodes — critical for stateful multi-pod workloads.
Exam trigger — "shared across instances / AZs" Whenever a scenario mentions multiple EC2 instances needing access to the same files simultaneously — especially across AZs — the answer is EFS. Not S3 (object storage, not mountable as a filesystem). Not EBS (single instance). EFS.
Windows = not EFS EFS is Linux/NFS only. Windows workloads needing shared file storage → Amazon FSx for Windows File Server (SMB protocol, NTFS filesystem, Active Directory integration). This distinction is tested frequently.
EFS — scenario practice
Q

10 EC2 instances in an Auto Scaling Group serve a WordPress site. When one instance uploads an image, other instances can't see it. Users get broken images depending on which instance they hit. Fix this with the least operational overhead.

A

Mount Amazon EFS on all instances and store the media library on EFS. All 10 instances (and any future ASG instances) mount the same filesystem — when one instance writes a file, all others see it immediately. EFS auto-scales, requires no capacity planning, and new ASG instances can be configured to mount at launch via user data.

The signal words: "shared files", "multiple instances", "ASG", "user uploads", "media library" → EFS. The alternative (S3 for media) is also valid but requires app code changes. EFS is a drop-in NFS mount — minimal code change.
Q

An EFS filesystem is getting expensive. Analysis shows 90% of files haven't been accessed in over 30 days. What's the cheapest fix that requires no application changes?

A

Enable EFS Intelligent-Tiering (or set a lifecycle policy to transition files to EFS-IA after 30 days of inactivity). Files are automatically moved to the cheaper IA tier with no application changes needed — the mount path stays the same, reads are transparent (just slightly slower and incur a per-GB retrieval fee). Storage cost drops by ~85% for the moved files.

IA = Infrequent Access. Storage is much cheaper but there's a per-access retrieval fee. If data is truly cold (accessed <once/month), the math strongly favours IA. Intelligent-Tiering automates the decision.
Q

Multiple ECS containers (across different tasks) need to share a writable persistent volume. You try using an EBS volume but only one task can mount it. What should you use instead?

A

EFS. EBS volumes can only be attached to one EC2 instance at a time (the host running the task), and even then only one ECS task can use it. EFS can be mounted by all containers across all ECS tasks simultaneously with full read/write access. Define the EFS volume in the ECS task definition and mount it as a container volume.

EBS = one instance, one task. EFS = many instances, many tasks. In Kubernetes (EKS) terms: EBS = ReadWriteOnce PVC. EFS = ReadWriteMany PVC. Any time "multiple containers" or "multiple pods" need shared writable storage → EFS.
Q

EC2 instances in us-east-1 can't mount the EFS filesystem even though the mount target exists and the instance is in the same VPC. What do you check first?

A

Check the Security Group on the EFS mount target. It must allow inbound TCP port 2049 (NFS) from the EC2 instance's security group. The most common EFS connectivity issue is this SG rule missing. Also verify: the mount target exists in the same AZ as the instance, and the instance's SG allows outbound on port 2049.

Port 2049 = NFS = EFS. Memorise this. Exam questions about EFS connectivity issues almost always resolve to: mount target SG not allowing 2049, or mount target not existing in the instance's AZ.

Decision guide — which to pick?
Q

Single EC2 instance running MySQL, PostgreSQL, or MongoDB

A

EBS gp3. Block storage, single instance owns the volume. High IOPS needed (busy OLTP)? → io2. The DB is the only thing reading/writing — no sharing needed.

Why not EFS: EFS is NFS — databases don't run on NFS due to locking semantics and latency. Block storage (EBS) is always the right answer for databases on EC2.
Q

Multiple EC2 instances in an ASG serving a CMS (WordPress, Drupal) that stores media uploads on the filesystem

A

EFS. All ASG instances mount the same EFS filesystem. A file written by one instance is immediately visible to all others. No sticky sessions, no sync needed.

Why not EBS: EBS can only attach to one instance. In an ASG, each instance would have its own isolated EBS volume — writes on instance A would never reach instance B.
Q

EKS pods need persistent shared storage with ReadWriteMany access — multiple pods writing simultaneously

A

EFS with the EFS CSI driver and a PVC set to ReadWriteMany. EBS via the EBS CSI driver only supports ReadWriteOnce — a single node. EFS supports RWX — all nodes, all pods, simultaneously.

K8s access modes: RWO (ReadWriteOnce) = EBS. RWX (ReadWriteMany) = EFS. This maps directly to how EBS is single-instance and EFS is concurrent multi-instance.
Q

Hadoop HDFS or Cassandra cluster on EC2 — needs highest possible I/O performance, application handles its own replication

A

Instance Store (NVMe). Millions of IOPS, sub-millisecond latency, no network hop. HDFS and Cassandra replicate data across nodes at the application layer — durability is handled in software. The storage just needs to be fast. Instance store is included in the instance price.

The trade-off: Instance store is the fastest storage on AWS but it's ephemeral — lost on stop/terminate/hardware failure. Only use it when the application itself provides durability (replication). Never for a single-instance database.
Q

Cross-region disaster recovery — database EBS volumes need to be recoverable in another region

A

Create EBS Snapshots (automated via AWS Backup or DLM) and configure cross-region snapshot copy to the DR region. In a disaster, create new EBS volumes from the snapshots in the DR region and attach to new instances. For EFS cross-region DR → use AWS DataSync.

Snapshots are the EBS DR primitive. They're incremental (only changed blocks after first), stored in S3 (durable), and can be copied cross-region. Always automate — never rely on manual snapshots for production.
Q

Multiple Windows EC2 instances need a shared network drive — users expect drive letters and Windows ACLs

A

Amazon FSx for Windows File Server. EFS is Linux/NFS only — Windows cannot mount it natively. FSx for Windows supports SMB protocol, NTFS filesystem, Windows ACLs, DFS namespaces, and Active Directory integration. It's the managed Windows file server on AWS.

The trap: The question says "shared file system" which sounds like EFS — but "Windows" is the disqualifier. EFS = Linux + NFS. FSx for Windows = Windows + SMB. This is tested frequently.

Exam Tips & Tricky Scenarios

High-yield patterns that appear consistently on the SAA-C03 exam. One detail in the scenario often changes the entire answer — read carefully.

EC2 exam patterns
Q

A company wants the lowest cost for a fault-tolerant video transcoding job that can be re-run if interrupted. Which purchase model?

A

Spot Instances. Fault-tolerant + restartable = Spot. Up to 90% cheaper. Use a Spot interruption handler to checkpoint state on the 2-minute warning.

Keywords → Spot: "fault-tolerant", "stateless", "can restart", "batch", "flexible start/end time". Keywords → On-Demand/Reserved: "cannot be interrupted", "stateful", "critical", "always running".
Q

A startup runs EC2 24/7 for 3 years and also uses Fargate for containers. Which commitment option gives the best overall discount?

A

Compute Savings Plan. It covers EC2, Fargate, and Lambda under one $/hr commitment. Standard RIs give slightly higher EC2 discounts (~72% vs ~66%) but don't cover Fargate. When the workload spans multiple compute types, Savings Plans win on flexibility.

Decision rule: Pure EC2 + want max discount → Standard RI. EC2 + Fargate/Lambda → Compute Savings Plan. Need to change instance family in future → Convertible RI or Compute SP.
Q

A distributed HPC simulation needs the absolute lowest network latency between 8 instances. Which placement group?

A

Cluster Placement Group. Packs all instances onto the same rack in the same AZ. Enables 10 Gbps+ between instances, lowest possible latency. Trade-off: if the rack fails, all instances fail together — no HA.

Placement group cheat sheet: Lowest latency / HPC → Cluster (same rack, same AZ). Max fault tolerance, few critical instances → Spread (max 7/AZ, different racks). Large distributed DB (HDFS/Cassandra) → Partition (isolated rack groups).
ENI exam patterns
Q

A primary EC2 database instance fails. The standby instance is ready. You need IP continuity — clients must reconnect to the same IP. What do you move?

A

Move the Elastic Network Interface (with its Elastic IP) from the failed instance to the standby. The IP is tied to the ENI — detach from failed, attach to standby. Clients reconnect to the same IP in seconds. No DNS change, no TTL wait.

vs Route 53 failover: R53 health checks + failover routing also work, but DNS TTL means 60s+ before clients switch. ENI move is layer 3 — near-instant. For strict RTO requirements, ENI is faster.
Q

An EC2 instance configured as a NAT gateway (NAT instance) isn't forwarding traffic for private subnet instances. Routes and SGs are correct. One setting is wrong — what is it?

A

Source/destination check is enabled. Disable it. When EC2 receives a packet that isn't addressed to it (which happens with every packet a NAT instance forwards), it drops it by default. Disabling src/dst check tells EC2 to forward those packets instead of dropping them.

This only affects NAT instances (DIY EC2 NAT). Managed NAT Gateway handles this automatically. On the exam, if they mention a NAT Instance (not NAT Gateway) and forwarding doesn't work → src/dst check.
EBS exam patterns
Q

An RDS-style workload on EC2 is experiencing high read/write latency. CloudWatch shows high VolumeQueueLength (I/O waiting). The volume is gp2 at 200 GB. What's the fix?

A

Migrate to gp3 with higher provisioned IOPS, or to io2 for mission-critical. On gp2 at 200 GB you only get 600 IOPS (3×200). On gp3, you can provision up to 16,000 IOPS independently of size. Also verify EBS-Optimized is enabled on the instance — without it, EBS I/O competes with network traffic.

VolumeQueueLength > 1 is the CloudWatch signal for I/O saturation. The fix is more IOPS (gp3/io2), not more storage. On gp2, those two are coupled — on gp3 they're independent.
Q

After terminating an EC2 instance, the root EBS volume was also deleted — taking the data with it. How do you prevent this?

A

Set Delete on Termination = false for the root volume in the Launch Template. By default it's true for root volumes (false for additional data volumes). This is a common production mistake — always set it explicitly in your launch templates and AMI configurations.

Default behaviour: Root volume → deleted on termination (default ON). Data volumes → preserved on termination (default OFF). Know these defaults and always override in production launch templates.
EFS exam patterns
Q

A company migrates an on-premises NFS share to AWS. 20 Linux EC2 instances across 3 AZs need concurrent access to the same files. What service replaces the NFS share?

A

Amazon EFS. It's a managed NFS (NFSv4) service that can be mounted by thousands of instances across multiple AZs simultaneously. Create one mount target per AZ, configure the instance SGs to allow port 2049, and mount using the EFS mount helper. Drop-in replacement for on-premises NFS.

"NFS", "shared files", "multiple Linux instances", "across AZs" → EFS every time. S3 is not a filesystem (no mount, no POSIX). EBS is single-instance. EFS is the NFS answer.
Q

An EFS file system stores 10 TB of data. Analytics show only 500 GB is accessed regularly — the rest hasn't been touched in 60 days. Monthly bill is high. What's the cheapest fix?

A

Enable EFS Lifecycle Management with a 30-day transition policy to move files to EFS-IA (Infrequent Access). The 9.5 TB of cold data moves to IA storage which costs ~85% less. Files are retrieved transparently when accessed — no application changes needed.

EFS-IA vs Standard: Standard ~$0.30/GB/month. IA ~$0.025/GB/month + $0.01/GB retrieval. For data accessed <once/month the savings are dramatic. Enable Intelligent-Tiering to automate the decision instead of setting a fixed day threshold.
