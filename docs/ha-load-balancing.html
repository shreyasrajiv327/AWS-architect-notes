§0

Why Load Balancers?

The core problems ELB solves
Single point of failure

Without a load balancer, one DNS entry points to one server. That server fails → entire app is down. A load balancer acts as the stable entry point and routes around failures automatically.

Horizontal scaling

Adding more servers ("scaling out") only helps if traffic can actually reach them. The load balancer distributes incoming connections across all healthy instances in a target group.

Health checking

ELB continuously pings each target on a configured path/port. Unhealthy targets are removed from rotation automatically — no manual intervention, no downtime from a bad deploy.

SSL/TLS termination

Terminate HTTPS at the load balancer. Backend instances only handle plain HTTP. One certificate to manage, not N. Use ACM (AWS Certificate Manager) for free auto-renewing certs.

Stickiness / session affinity

Option to pin a user's requests to the same backend instance for the duration of a session. Cookie-based (app or duration cookie). Useful for stateful legacy apps that can't share session state.

Cross-zone load balancing

Distribute traffic evenly across all registered targets in all AZs — not just the AZ the request landed in. Prevents hot AZs when instance counts differ per AZ.

Managed service — this matters for the exam AWS manages the ELB fleet. You don't patch, scale, or maintain the load balancer nodes. AWS guarantees high availability of the ELB itself. The only thing you manage is config: listeners, target groups, rules, health checks. "Undifferentiated heavy lifting" — classic Stephane framing.
Health checks — the foundation of everything
How it worksELB sends a request to each target on a specified protocol, port, and path every N seconds. If the response is not a 200 (or a configured success range), the target is marked unhealthy and taken out of rotation.
Key settings
  • Protocol: HTTP, HTTPS, TCP, gRPC
  • Path: e.g. /health — must return 200 fast
  • Interval: 5–300s (default 30s)
  • Healthy threshold: consecutive successes to mark healthy (default 3)
  • Unhealthy threshold: consecutive failures to mark unhealthy (default 3)
  • Timeout: how long to wait per check (must be < interval)
Production tipYour /health endpoint should be lightweight and truthful. It should check DB connectivity and critical dependencies — not just return 200 blindly. A 200 from a broken app is worse than a 500 (ELB will keep sending traffic to a broken instance).
§1

Application Load Balancer

ALB — Layer 7 (HTTP/HTTPS/gRPC/WebSocket)

ALB understands the content of the request — URL path, host header, query strings, HTTP method, source IP. It makes routing decisions based on this context. This is the right choice for virtually all modern web applications and microservices. It's the most feature-rich AWS load balancer.

Routing rules — the superpower of ALB

ALB listener rules are evaluated in priority order (lowest number first). The first matching rule wins. Each rule has conditions and an action (forward, redirect, fixed-response, authenticate).

Listener: HTTPS :443 — Rule evaluation order
P: 1 Host: api.example.com → Forward → api-service TG (ECS containers, port 8080)
P: 2 Path: /images/* → Forward → static-assets TG (S3 / CDN origin instances)
P: 3 Query: ?version=v2 → Forward → v2-canary TG (weighted: 10% new, 90% stable)
P: 4 HTTP: GET + /admin/* → Authenticate (Cognito) → Forward → admin-service TG
Default Everything else → Forward → web-app TG (main application)
Condition types
  • Host-based: Route subdomains to different services (api.* → API, app.* → frontend)
  • Path-based: Route URL prefixes (/api/* → API service, /checkout/* → checkout microservice)
  • HTTP header: Match any request header value (e.g. custom feature-flag header)
  • HTTP method: Route GET vs POST differently (e.g. reads vs writes to different clusters)
  • Query string: Route based on query params (?platform=mobile → mobile API)
  • Source IP: Route internal vs external IPs to different backends
Action types
  • Forward: Send to one or more target groups (with weights for canary releases)
  • Redirect: HTTP → HTTPS, domain changes, path changes (301/302)
  • Fixed response: Return a static response directly from ELB (useful for maintenance pages or health endpoints)
  • Authenticate: Integrate with Cognito or OIDC provider before forwarding
Target groups
Target typesEC2 instances (by instance ID), IP addresses (any IP — even on-premises via Direct Connect/VPN), Lambda functions, other ALBs. Each target group has its own health check config.
Weighted routingOne listener rule can forward to multiple target groups with weights. e.g. 90% to stable, 10% to canary. No DNS TTL waiting — change weights live. Essential for blue/green and canary deployments.
ECS integrationECS registers and deregisters tasks directly with target groups. Dynamic port mapping — multiple tasks on one EC2 host, each on a different ephemeral port. The TG handles it automatically.
Lambda targetsInvoke Lambda directly via ALB. ALB converts HTTP request to JSON event. Lambda returns a JSON response which ALB converts back to HTTP. Fully serverless request handling with ALB routing logic in front.
Exam — ALB cannot see client IP directly at the instance ALB terminates the connection and creates a new one to the backend. The backend sees the ALB's private IP as source, not the client IP. The real client IP is forwarded in the X-Forwarded-For header. Also: X-Forwarded-Port (original port) and X-Forwarded-Proto (original protocol). Your app must read these headers for client IP-based logic.
ALB key features
WebSocket supportNative. ALB upgrades the HTTP connection to WS and maintains the persistent connection. No special config — just target the backend port. Crucial for real-time apps (chat, live dashboards).
HTTP/2Between client and ALB only. ALB converts to HTTP/1.1 for backend communication (unless backend also supports H/2 and you enable it). Improves browser performance without requiring backend changes.
gRPCALB understands gRPC — it can route, health-check, and load-balance gRPC services. Use for microservices that communicate over gRPC.
Sticky sessionsCookie-based. ALB inserts an AWSALB cookie. Duration-based (you set TTL) or Application-based (your app sets the cookie, ALB reads it). Stickiness breaks even distribution — avoid if possible, use shared session stores (ElastiCache) instead.
SSL/TLS + SNIALB supports Server Name Indication (SNI) — multiple SSL certs on a single ALB listener (one cert per domain). No need for multiple ALBs for multi-tenant or multi-domain apps. Certificates via ACM (free auto-renew).
Auth integrationNative Cognito User Pool integration and OIDC (Google, Facebook, Okta). Authenticate at the load balancer — zero code changes in your app. Returns user identity in headers after auth.
Access logsDetailed logs per request to S3: client IP, latency, request path, backend response code, SSL cipher, etc. Essential for debugging and forensics. Not enabled by default — turn on in production.
Production pattern — ALB as microservices API gateway One ALB, multiple target groups. Host-based rules route api.app.com to one ECS service, app.com to another. Path-based rules subdivide further (/auth/* → auth service, /orders/* → orders service). Add Cognito auth at the ALB for protected routes. Weighted forwarding for canary deploys. This replaces a lot of what people build custom API gateways for.
ALB is not for ultra-low-latency or non-HTTP ALB adds ~1–5ms latency (it's a full proxy). For UDP, raw TCP, sub-millisecond latency, or non-HTTP protocols — use NLB. ALB also doesn't support Elastic IPs or a static IP (you get a DNS name that resolves to changing IPs).
§2

Network Load Balancer

NLB — Layer 4 (TCP / UDP / TLS)

NLB operates at the transport layer. It does not look inside the packet — it just routes based on IP and port. This makes it blazing fast — millions of requests per second, sub-millisecond latency, no added overhead. The right choice when you need raw performance, static IPs, or non-HTTP protocols.

NLB key characteristics
PerformanceHandles millions of requests per second. Latency as low as ~100µs. NLB nodes don't terminate connections — they pass TCP/UDP flows through with minimal processing. AWS recommends NLB for extreme throughput workloads.
Static IPsEach NLB node in an AZ gets a fixed Elastic IP. This is the most important NLB feature for enterprise and compliance scenarios: firewall rules, IP allowlists, third-party integrations that require stable IPs. ALB only gives you a DNS name.
Client IP preservationNLB passes the original client IP directly to the backend (unlike ALB which proxies and adds X-Forwarded-For). The backend sees the real source IP on the socket. For TCP targets, configure security groups to allow the client IPs. For instance targets in same VPC, client IP visible. For IP targets, configure properly.
ProtocolsTCP, UDP, TCP_UDP, TLS. Covers everything ALB doesn't: DNS (UDP 53), gaming (UDP), IoT protocols, raw TCP services, custom binary protocols.
TLS terminationNLB can terminate TLS (like ALB). Also supports TLS passthrough — forward encrypted traffic straight to backend without decrypting. Use passthrough when end-to-end encryption to the app is required (compliance, mTLS).
Cross-zone LBDisabled by default (unlike ALB where it's on by default). When disabled: traffic in AZ-A only goes to targets in AZ-A. When enabled: traffic distributes across all AZs. Data transfer charges apply when cross-zone is enabled on NLB.
Zonal isolationBecause NLB has per-AZ static IPs, you can route AZ-specific traffic explicitly. Powerful for latency-sensitive architectures where you want to minimize cross-AZ hops.
NLB target groups
Target typesEC2 instances, IP addresses, ALB. Yes — you can put an ALB behind an NLB. This gives you NLB's static IPs at the entry point with ALB's routing capabilities behind it. Classic pattern for enterprise use.
NLB → ALB patternNLB provides: fixed IPs, ultra-fast entry, TLS passthrough option. ALB provides: host/path routing, Cognito auth, header inspection. Together: static IP entry + intelligent routing. Very common for B2B APIs where clients need stable IPs to allowlist.
Health checksTCP, HTTP, HTTPS. For TCP: just checks if the port is open (connection established). For HTTP/HTTPS: checks response code. TCP health checks are faster and simpler but less meaningful — prefer HTTP/HTTPS if your app supports it.
Security groupsNLB itself doesn't have a security group (it passes traffic transparently). EC2 instances in NLB target groups must allow the client IPs directly in their SGs (or 0.0.0.0/0 for public NLB). This is a common gotcha.
Exam — NLB has no security group ALB has a security group. NLB does NOT. Traffic flows through NLB transparently. Backend EC2 instances must allow client IPs (or 0.0.0.0/0 for internet-facing). If a question says "EC2 instances behind NLB are not receiving traffic" — check the instance SG, not an NLB SG (it doesn't exist).
Exam — static IP = NLB Whenever the scenario mentions: "clients need to whitelist the load balancer IP", "firewall rules require a fixed IP", "third-party service requires a static endpoint" → the answer is NLB (with Elastic IPs per AZ). ALB only provides a DNS hostname with changing IPs.
Production pattern — NLB for gaming / IoT Game servers, IoT MQTT brokers, VoIP — all UDP or custom TCP. NLB handles UDP natively at millions of connections per second. Configure UDP listeners, set health checks to TCP (since UDP is connectionless and harder to health-check), and use IP target groups for maximum flexibility.

Gateway Load Balancer

GWLB — Layer 3 (IP Packets)

GWLB is fundamentally different from ALB and NLB. It doesn't load-balance your application traffic to make it more available. Instead, it invisibly inserts a fleet of network appliances into the traffic path — firewalls, IDS/IPS, deep packet inspection tools — before traffic reaches your application. Think of it as a transparent bump-in-the-wire at scale.

How GWLB works — the traffic flow
The problem it solvesYou want all ingress (and/or egress) traffic to pass through a 3rd-party security appliance (Palo Alto, Fortinet, Check Point, etc.) for inspection. Without GWLB, this is complex — you'd need to manually route traffic to the appliance and back. GWLB automates this transparently.
GENEVE protocolGWLB uses the GENEVE tunnelling protocol (port 6081) to encapsulate original packets and send them to appliance instances. The appliance sees the original src/dst IPs (not tunnel IPs), inspects the packet, and returns it. GWLB then forwards to the actual destination. All transparent to both sender and receiver.
Traffic flow
  • Ingress: Internet → GWLB → (encapsulate) → security appliance fleet → (pass/drop) → (decapsulate) → your application
  • Egress: Your app → GWLB → security appliance fleet → Internet (for outbound inspection)
  • Traffic is returned to GWLB for forwarding — the appliance doesn't route it itself
Gateway Load Balancer Endpoint (GWLBE)A VPC endpoint (powered by PrivateLink) that you place in your VPC's route tables. Traffic destined for your app is redirected to the GWLBE → GWLB → appliances → back to app. The app sees nothing different.
Appliance target groupThe security appliances (EC2 instances running firewall software) register in the GWLB target group. GWLB load balances across the appliance fleet for HA and scalability. Add more instances to scale inspection throughput.
Exam — GWLB is for transparent network inspection, not app load balancing If the question mentions: 3rd-party firewalls, IDS/IPS, deep packet inspection, traffic inspection before reaching the app, inserting security appliances into traffic path — the answer is GWLB. It operates at Layer 3 (IP), not Layer 7 (HTTP) or Layer 4 (TCP). It uses GENEVE protocol on port 6081.
Production pattern — centralized security inspection VPC Typical enterprise architecture: a dedicated "security VPC" with a GWLB and your appliance fleet. All spoke VPCs route internet-bound and inbound traffic through the security VPC via Transit Gateway + GWLB endpoints. Centralized inspection, single fleet to manage, scales independently of applications.

Decision guide — which load balancer?
Standard web application behind HTTPS with multiple microservices (e.g. /api/* → API, /app/* → frontend)
→ ALB. Path-based routing is native. Add HTTPS listener, ACM cert, path rules per service.
B2B API — partner companies need to whitelist your load balancer's IP in their firewall
→ NLB. Static Elastic IPs per AZ. Partners allowlist those IPs. Put ALB behind NLB if you also need content routing.
Online multiplayer game using UDP for real-time state synchronization
→ NLB. Only load balancer that supports UDP. Sub-millisecond latency, millions of concurrent connections.
Route traffic to different environments based on subdomain (dev.app.com vs prod.app.com)
→ ALB with host-based routing rules. One ALB, two target groups, one rule per subdomain.
Security compliance requires all traffic to pass through Palo Alto firewall before reaching any EC2 instance
→ GWLB. Insert the firewall appliance fleet transparently. Traffic flows through without the app knowing.
Multi-tenant SaaS — each tenant has their own subdomain and SSL certificate
→ ALB with SNI. Multiple SSL certs on one listener. Host-based rules route each subdomain to the correct tenant's target group.
Internal service that needs end-to-end TLS — traffic must be encrypted all the way to the backend
→ NLB with TLS passthrough. NLB forwards encrypted traffic without decrypting it. Backend terminates TLS itself.
Canary deployment — send 5% of traffic to new version, 95% to stable
→ ALB with weighted target groups on a forwarding rule. Set weights 95/5. Adjust live without DNS changes.

Exam Traps

Cross-zone load balancing defaults ALB: enabled by default, no extra charge. NLB: disabled by default, charged when enabled. GWLB: disabled by default, charged when enabled. This appears in cost-optimization questions — "why is there cross-AZ data transfer charge?" → NLB with cross-zone enabled.
Connection draining / deregistration delay When a target is deregistered (or marked unhealthy), ELB stops sending new requests but keeps in-flight requests alive for up to the deregistration delay (default 300s, can be 0–3600s). This is called Connection Draining on Classic LB and Deregistration Delay on ALB/NLB. Exam will ask about graceful shutdowns — this is the mechanism.
Slow start mode (ALB only) New targets get gradually increasing traffic over a warm-up period (30–900s) instead of immediately receiving their full share. Use this for targets that need time to warm up (JVM apps with JIT compilation, cache-heavy apps). Exam may present it as "how to avoid overwhelming new instances".
ALB listener redirect — HTTP to HTTPS Create an HTTP :80 listener with a Redirect action to HTTPS :443. This is the standard pattern. The redirect happens at the ALB — no code in your app. Common exam question about enforcing HTTPS.
ALB fixed response You can return a static response (status code + body) directly from ALB without hitting any backend. Use for: maintenance mode pages, returning 403 for unauthorized paths, health checks that always return 200. Exam presents this as "return a custom error page without reaching the backend".
GWLB port 6081 / GENEVE GWLB uses the GENEVE protocol on UDP port 6081 to tunnel packets to appliances. Security group on the appliance EC2 must allow port 6081 from the GWLB. This is a very specific detail Stephane covers — it does appear in questions.
